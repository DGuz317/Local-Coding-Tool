## False Positives

- `tests/fixtures/dogfood/js-ts-workspace/packages/app/src/index.ts` imports `@dog/lib`, which is a local workspace package in `packages/lib/package.json`; the JavaScript import table classifies that import as `third_party` with `Resolution=external`.
- `tests/fixtures/dogfood/mixed-docs-config/Makefile` classifies `make verify` with purpose `unknown`; this is conservative but less useful than recognizing it as verification.

## False Negatives

- JS/TS workspace path aliases from root `tsconfig.json` are not connected to package import resolution for `@dog/lib`.
- The mixed docs/config fixture records commands only from `Makefile`; equivalent declarative command strings in `config/service.yaml` are not promoted into command facts.
- Impact analysis and suggested reading order remain shallow for docs/config-only repositories because no semantic source graph exists there.

## Known Limitations

- Package Boundary behavior is partial: config indexing sees local package roots, but JS/TS import resolution does not yet use workspace package metadata as a local resolution source.
- Makefile command purpose detection is intentionally shallow and should not infer safe auto-execution.
- RepoLens does not emulate TypeScript compiler, package manager, bundler, or framework resolution.
- RepoLens does not execute candidate commands found in package scripts, Makefiles, or documentation.

The package/workspace finding is documented here and remains a #50 follow-up. It is not promoted to a new P0 bug from this dogfood run because the graph stays safe, marks commands as not run, and does not produce corrupt artifacts or source disclosure.
