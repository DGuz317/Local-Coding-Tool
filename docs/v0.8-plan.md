# RepoLens v0.8 Plan

Status: draft planning document, revised after `docs/repolens-v0.8-plan-review.md`.

## Recommendation

For **v0.8**, ship a narrow **AI Proposal Layer**. Do not ship a broad AI review suite, chatbot, agent runner, or active workflow system.

```text
v0.8 = optional AI proposals over bounded RepoLens metadata
v0.9 = approved active workflows and command/file-write surfaces
```

The first AI release should prove the trust boundary before adding higher-risk proposal types.

Recommended v0.8 scope:

```text
AI Proposal contract
+ AI provider boundary
+ metadata-only AI input packer
+ Context Pack Summary Proposal
+ Architecture Explanation Proposal
+ Patch Plan Proposal
+ AI Artifact Safety Audit
```

Defer until after initial dogfood:

```text
- Convention Proposal
- Local Change Review Proposal
- Deterministic Fact Opportunity Proposal
- Test Gap Proposal
- Ownership Hypothesis
- Task Intent Proposal
```

## v0.8 Goals

v0.8 should prove:

1. AI is disabled by default and never makes hidden network calls.
2. AI inputs are bounded RepoLens metadata by default, not whole source files.
3. AI outputs are labeled AI Proposals with provider/model provenance and reproducibility metadata.
4. AI Proposals never become deterministic graph facts by direct promotion.
5. Default Context Packs, graph hashes, resolver behavior, package ownership, and MCP read-only behavior remain deterministic without AI.
6. RepoLens can summarize Context Packs, explain architecture, and propose patch plans as optional read-only AI outputs.
7. Saved AI Proposal artifacts are explicit opt-in and pass Artifact Safety Audit.

## Non-Goals

Do not include these in v0.8:

```text
- default AI enablement
- hidden external provider calls
- provider fallback
- hidden default model
- silent network access
- telemetry
- embeddings or vector search
- source-code upload by default
- source-mirroring AI artifacts
- AI-generated graph facts promoted as truth
- MCP write tools
- command execution
- automatic test execution
- apply-ready patch/diff execution
- local change review proposal as a release feature
- convention proposal as a release feature
- deterministic fact opportunity as a release feature
- ownership or task-intent hypotheses as release features
- GitHub/GitLab bot comments
- remote PR mutation
- hosted service APIs
- browser UI
```

Patch plans are allowed as AI Proposal text. Applying a patch, writing files, producing apply-ready diffs, or running commands belongs to the v0.9 Active Workflow layer.

## Preconditions

v0.8 depends on the earlier code-intelligence foundation being stable enough to feed bounded AI inputs.

Required or strongly preferred before implementation:

```text
- Context Pack schema and ranking contracts are stable.
- MCP response envelope carries trust, freshness, limits, and recoverable problems.
- Resolver Evidence Taxonomy and Relationship Candidate language exist.
- Artifact Safety Audit exists or is close enough to extend.
- Context Pack Evaluation can prove default deterministic behavior does not change.
- Candidate Verification Commands are represented as not-run metadata.
```

Branch-aware freshness and local diff review metadata are useful, but they should not block v0.8 because Local Change Review Proposal is deferred.

If a semantic fact type is unavailable, AI Proposal output must say so instead of implying certainty. For example, no CFG means no control-flow certainty; no framework route hint means no route-ownership claim.

## Core Contract

### AI Proposal Layer

The AI Proposal Layer is optional. It consumes bounded RepoLens metadata and emits labeled proposals.

```text
Input:
  graph facts
  Context Pack metadata
  structural summaries
  relationship candidates
  graph quality warnings
  related tests when present
  candidate verification commands marked run=false

Output:
  AI Proposals
```

It must not mutate:

```text
- graph artifacts
- Canonical Graph Hash
- deterministic Context Pack IDs
- default Context Pack ranking
- resolver behavior
- package ownership
- branch state
- user files
```

### AI Proposal Shape

Every AI Proposal should carry enough metadata for callers to understand trust boundaries and reproduce the deterministic input boundary without storing raw AI input.

Required fields should include the equivalent of:

```text
kind
proposal_schema_version
provider
model
input_boundary
source_disclosure
context_pack_id or target refs when applicable
evidence refs
confidence label
warnings
limitations
input_packer_version
redaction_policy_version
input_digest
context_pack_version when applicable
graph_schema_version
canonical_graph_hash
```

The `input_digest` should hash the bounded packed metadata sent to the provider. It should allow auditability without persisting raw prompt/input content.

### Initial Proposal Kinds

Ship only these proposal kinds in v0.8:

```text
context_pack_summary
architecture_explanation
patch_plan
```

Keep these names reserved for post-v0.8 work:

```text
convention_proposal
local_change_review_proposal
deterministic_fact_opportunity
test_gap_proposal
task_intent_proposal
ownership_hypothesis
```

The most important naming choice is `deterministic_fact_opportunity`, not `graph_fact_proposal`. It avoids implying that AI has created or discovered a trusted graph fact.

### AI Provider Boundary

AI is disabled by default.

Provider rules:

```text
No configured provider means no AI call.
No provider fallback.
No hidden default model.
No silent network access.
```

External providers require explicit configuration. Provider configuration must remain local-only.

Credential rules:

```text
- Environment variables may supply credentials.
- RepoLens config may reference credential environment variable names.
- RepoLens config must not store credential values.
- Credentials must not appear in artifacts.
- Credentials must not appear in MCP responses.
- Credentials must not appear in AI Proposal metadata.
- Provider error payloads must be redacted before returning to CLI or MCP.
```

Every AI Proposal must expose provider/model provenance. When AI is disabled or unconfigured, RepoLens should return a structured disabled/unavailable result, not fall back to hidden defaults.

### AI Input Boundary

Default AI inputs are bounded metadata, not raw source.

Allowed by default:

```text
- repo-relative POSIX paths
- symbol names
- package identities
- relationship labels
- evidence labels
- line ranges
- graph warnings
- risk signal metadata
- Context Pack item reasons
- candidate commands marked run=false
- bounded structural summaries
```

Not allowed by default:

```text
- whole source files
- function/class bodies
- raw comments
- raw secret-like values
- large raw document excerpts
- raw agent guidance text
- raw configuration dumps
- provider credential values
- raw provider error payloads
```

If a future feature needs source text, it must use an explicit scoped approval path and must not persist raw source text as AI artifact content.

### Persistence Boundary

AI Proposals are returned ephemerally by default.

Persistence rules:

```text
- AI Proposal artifacts are not saved by default.
- Persisting AI Proposal artifacts requires an explicit --save flag or equivalent option.
- Saved proposals must pass Artifact Safety Audit.
- Saved proposals must store proposal metadata and bounded output, not raw source-bearing AI input.
- Saved proposals must not store credentials or unredacted provider errors.
```

This prevents `.repolens` from accumulating AI-generated metadata unexpectedly.

## User-Facing v0.8 Slices

### 1. Context Pack Summary Proposal

A user can request an optional AI summary for an existing Context Pack.

Expected behavior:

```text
- Produces AIProposal(kind=context_pack_summary).
- Uses bounded Context Pack metadata.
- Is distinct from deterministic Structural Summary.
- Does not change Context Pack ID or ranking.
- Separates deterministic evidence from AI interpretation.
- Lists limitations and missing fact types.
```

### 2. Architecture Explanation Proposal

A user can ask RepoLens to explain a repository area, package boundary, file group, symbol target, or Context Pack.

Expected behavior:

```text
- Produces AIProposal(kind=architecture_explanation).
- Explains responsibilities and neighboring areas.
- Cites deterministic evidence refs.
- Surfaces unresolved candidates and graph quality warnings.
- Includes limitations.
- Does not create ownership, dependency, route, runtime behavior, or graph facts.
```

### 3. Patch Plan Proposal

A user can request a read-only implementation plan for a task.

Expected output:

```text
AIProposal(kind=patch_plan)
  goal
  target files to inspect
  suggested edit sequence
  related tests to update or inspect
  docs/config risks
  candidate verification commands marked run=false
  graph evidence refs
  limitations
```

Expected behavior:

```text
- Plan cannot be applied by RepoLens in v0.8.
- No command execution occurs.
- No file writes occur.
- No apply-ready diff is produced as a v0.8 behavior.
```

## Deferred Proposal Types

The following proposal types are useful but should ship only after the v0.8 AI Proposal contract has dogfood evidence:

```text
convention_proposal
local_change_review_proposal
deterministic_fact_opportunity
test_gap_proposal
task_intent_proposal
ownership_hypothesis
```

Reasons to defer:

```text
- They can overclaim unsupported semantics.
- They can imply graph mutation or ownership certainty.
- They can create noisy findings before proposal quality is understood.
- They need stronger dogfood evidence and fixture design.
```

The future promotion path for deterministic fact opportunities remains:

```text
Deterministic Fact Opportunity Proposal
  -> human review
  -> distilled fixture
  -> deterministic extractor/resolver implementation
  -> tests
  -> stable graph fact only after contract change if needed
```

## Interface Direction

Prefer one narrow internal capability over many one-off commands.

Recommended internal service shape:

```text
create_ai_proposal(kind, target, context_pack_id, task, options)
```

Recommended CLI shape:

```bash
repolens ai proposal --kind context-pack-summary --context-pack <id>
repolens ai proposal --kind architecture-explanation --target <path-or-symbol>
repolens ai proposal --kind patch-plan --task "..."
```

Recommended MCP shape:

```text
create_ai_proposal
```

Keep MCP narrow and read-only. Do not add separate MCP tools for every proposal type until the service shape has proven stable.

The MCP tool may return proposal data, disabled/unconfigured provider states, and validation errors. It must not run commands, write files, post remote comments, apply patches, or persist proposals unless an explicit save-capable non-default surface is approved later.

## Evaluation

v0.8 evaluation should remain deterministic where possible.

Required checks:

```text
- default no-AI behavior is unchanged
- Context Pack IDs are unchanged when AI is disabled
- Canonical Graph Hash is unchanged by AI Proposal generation
- AI input packer is deterministic for the same graph/context input
- AI input digest is stable for equivalent packed input
- AI Proposal envelopes satisfy schema contracts
- Artifact Safety Audit covers saved AI Proposal output
```

Dogfood questions:

```text
- Did Context Pack summaries help an assistant choose better first reads?
- Did architecture explanations cite useful evidence refs?
- Did patch plans identify plausible target files and related tests?
- Did any AI output overclaim beyond available graph evidence?
- Did any AI output imply command execution, file writes, graph mutation, or remote posting?
```

Dogfood findings should become:

```text
- distilled fixtures
- documented limitations
- future deterministic extractor/resolver issues
```

They should not become hosted telemetry.

## v0.8 Release Gate

Before cutting v0.8, require the relevant focused gate plus the normal release gate when feasible:

```bash
uv run pytest
uv run ruff check .
uv run ruff format --check .
uv run mypy src/repolens
uv run repolens evaluate-context --json
uv build --out-dir /tmp/repolens-dist --clear
```

Also require:

```text
- AI disabled-by-default smoke test
- AI provider unconfigured smoke test
- AI provider configured smoke test using a fake/local test provider
- AI provider error redaction test
- AI input packer disclosure/redaction tests
- AI input digest determinism test
- AI Proposal schema/contract tests
- Artifact Safety Audit including saved AI Proposal fixtures
- Context Pack Summary fixture
- Architecture Explanation fixture
- Patch Plan Proposal fixture
- Documentation updated for AI boundaries, persistence behavior, and non-goals
```

If a command cannot be run, the release note must state why and what evidence replaced it.

## Bottom Line

v0.8 should introduce AI as a narrow, auditable, metadata-only read surface first.

```text
AI may summarize, explain, and propose patch plans.
The trusted graph stores deterministic evidence and provenance.
Local review, convention inference, deterministic fact opportunities, ownership hypotheses, and task-intent proposals wait until after dogfood.
Active file writes and command execution wait for v0.9.
```
